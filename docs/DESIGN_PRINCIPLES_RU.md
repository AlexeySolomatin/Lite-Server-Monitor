
# Lite Server Monitor (LSM)

## Принципы разработки

Версия: 1.0

Статус: Draft

---

# Назначение

Данный документ определяет основные принципы разработки проекта **Lite Server Monitor**.

Все новые компоненты, модули и изменения должны соответствовать данным принципам.

---

# 1. Keep It Simple (KISS)

Основной принцип проекта — простота.

Если задачу можно решить простым способом без потери надежности, следует выбрать именно его.

Проект не должен усложняться без объективной необходимости.

---

# 2. Модульность

Каждый модуль выполняет только одну задачу.

Например:

- Disk Monitor
- SMART Monitor
- RAID Monitor
- UPS Monitor

Модули должны быть максимально независимыми.

---

# 3. Один модуль — одна ответственность

Каждый модуль отвечает только за свою область.

Например:

Disk Monitor не должен выполнять проверку SMART.

SMART Monitor не должен отправлять уведомления самостоятельно.

Это делает систему более простой для сопровождения и тестирования.

---

# 4. Единая система уведомлений

Все уведомления проходят через единый модуль.

```
monitor_notify.sh
```

Модули никогда не отправляют Telegram или Email напрямую.

---

# 5. Общая библиотека

Повторяющийся код не должен копироваться между файлами.

Общие функции размещаются в каталоге:

```
lib/
```

Если одна и та же функция используется более одного раза — она переносится в библиотеку.

---

# 6. Единый CLI

Все действия пользователя выполняются через одну команду.

```
lsm
```

Например:

```bash
lsm install
lsm update
lsm doctor
lsm report
lsm status
```

Пользователь не должен запускать внутренние скрипты вручную.

---

# 7. Единая структура каталогов

Проект использует стандартную структуру Linux (FHS).

```
/opt/lsm
```

Файлы программы.

```
/etc/lsm
```

Конфигурация.

```
/var/log/lsm
```

Логи.

```
/var/lib/lsm
```

State-файлы.

```
/run/lsm
```

Временные данные.

---

# 8. Один источник истины

Все пути, константы и основные параметры определяются только в одном месте.

```
lib/common.sh
```

Использование абсолютных путей внутри модулей не допускается.

---

# 9. Идемпотентность

Повторный запуск установщика не должен:

- нарушать существующую конфигурацию;
- удалять пользовательские данные;
- создавать дублирующиеся службы;
- вызывать ошибки.

Установщик должен корректно работать как при первой установке, так и при повторном запуске.

---

# 10. Минимум зависимостей

Проект должен использовать только действительно необходимые зависимости.

Предпочтение отдается стандартным утилитам Linux.

Использование тяжелых внешних компонентов допускается только при наличии веских причин.

---

# 11. systemd First

Для автоматизации используются:

- systemd services;
- systemd timers.

Использование cron не планируется.

---

# 12. Конфигурация отдельно от кода

Все пользовательские настройки располагаются только в:

```
/etc/lsm
```

Исходный код проекта никогда не изменяется пользователем.

---

# 13. Безопасность по умолчанию

Настройки по умолчанию должны быть безопасными.

Если существует несколько вариантов поведения, предпочтение отдается наиболее безопасному.

---

# 14. Читаемость важнее краткости

Код должен быть понятным.

Допускается написать несколько дополнительных строк, если это улучшает читаемость.

---

# 15. Логирование

Все сообщения пользователю выводятся через общие функции логирования.

Например:

```bash
log_info
log_warn
log_error
log_success
```

Использование `echo` для служебных сообщений не рекомендуется.

---

# 16. Обработка ошибок

Все ошибки должны:

- иметь понятное описание;
- корректно завершать выполнение при необходимости;
- не оставлять систему в неконсистентном состоянии.

---

# 17. Совместимость

Все компоненты должны тестироваться на поддерживаемых версиях Ubuntu и Debian.

Использование специфичных для конкретного дистрибутива возможностей должно быть сведено к минимуму.

---

# 18. Документирование

Каждый новый модуль должен иметь:

- понятное название;
- краткое описание назначения;
- комментарии только там, где они действительно необходимы.

Документация должна обновляться одновременно с добавлением нового функционала.

---

# 19. Размер файлов

Следует избегать создания слишком больших файлов.

Приблизительные рекомендации:

- до 300 строк — отлично;
- до 500 строк — допустимо;
- более 500 строк — рекомендуется разделить на несколько модулей.

---

# 20. Развитие проекта

Новый функционал должен расширять существующую архитектуру, а не нарушать её.

Перед добавлением новой возможности необходимо оценить:

- необходимость;
- влияние на существующую архитектуру;
- сложность сопровождения;
- соответствие принципам данного документа.

---

# Заключение

Lite Server Monitor создается как простой, надежный и легко сопровождаемый инструмент.

Если при разработке возникает выбор между более сложным и более простым решением, предпочтение должно отдаваться простому, если оно не ухудшает надежность и функциональность проекта.
