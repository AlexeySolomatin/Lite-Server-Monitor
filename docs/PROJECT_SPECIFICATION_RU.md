# Lite Server Monitor (LSM)

## Спецификация проекта

Версия: 1.0

Статус: Draft

---

# 1. О проекте

**Lite Server Monitor (LSM)** — это легкий, модульный инструмент мониторинга и базовой защиты серверов под управлением Linux.

Проект предназначен для домашних серверов, NAS, небольших офисных серверов, серверов печати, систем видеонаблюдения и других небольших инфраструктур, где необходим простой, надежный и практически не требующий обслуживания мониторинг.

Основная идея проекта — предоставить удобный инструмент, который можно установить за несколько минут и использовать без постоянного сопровождения.

---

# 2. Цели проекта

Lite Server Monitor создается со следующими целями:

- простой процесс установки;
- минимальное количество зависимостей;
- низкое потребление системных ресурсов;
- модульная архитектура;
- удобная настройка;
- единый интерфейс управления;
- надежная работа в автоматическом режиме;
- соответствие стандартной структуре Linux.

---

# 3. Основные принципы

Проект строится вокруг следующих принципов:

- Простота важнее количества функций.
- Каждая функция должна решать одну конкретную задачу.
- Минимум внешних зависимостей.
- Безопасные настройки по умолчанию.
- Повторный запуск установщика не должен нарушать работоспособность системы.
- Все компоненты должны быть максимально независимыми друг от друга.

---

# 4. Поддерживаемые операционные системы

Поддерживаются:

- Ubuntu Server 22.04 LTS
- Ubuntu Server 24.04 LTS
- Debian 12

В дальнейшем список может быть расширен.

```
Lite-Server-Monitor/

```

---

# 7. Командный интерфейс

После установки управление системой выполняется через единую команду:

```bash
lsm
```

Основные команды:

```text
lsm install

lsm update

lsm uninstall

lsm doctor

lsm status

lsm report

lsm version
```

Пользователь не должен запускать внутренние скрипты напрямую.

---

# 8. Модульная архитектура

Каждый модуль отвечает только за одну задачу.

Например:

```
Disk Monitor

SMART Monitor

RAID Monitor

Temperature Monitor

UPS Monitor

Login Monitor

Fail2Ban Monitor
```

Модули должны быть максимально независимыми.

---

# 9. Мониторинг

Проект включает следующие категории мониторинга.

## Система

- CPU
- RAM
- Load Average
- Temperature

## Хранилище

- Disk Usage
- SMART
- RAID

## UPS

- APC UPS
- apcupsd

## Безопасность

- SSH Login Monitor
- Failed Login Statistics
- Fail2Ban Monitor

---

# 10. Система уведомлений

Поддерживаются:

- Telegram
- Email (SMTP / msmtp)

Все уведомления отправляются через единый модуль.

Это обеспечивает единый формат сообщений и упрощает сопровождение проекта.

---

# 11. Ежедневый отчет

LSM формирует ежедневый отчет, содержащий:

- состояние дисков;
- SMART;
- RAID;
- CPU;
- RAM;
- температуру;
- состояние UPS;
- статистику входов;
- статистику Fail2Ban;
- краткую сводку безопасности.

---

# 12. Конфигурация

Все пользовательские настройки располагаются в каталоге:

```
/etc/lsm
```

Предполагается использование нескольких конфигурационных файлов.

Например:

```
config.conf

modules.conf

notifications.conf

thresholds.conf

secrets.conf
```

Каждый файл отвечает только за свою область настроек.

---

# 13. Логирование

Все журналы располагаются в:

```
/var/log/lsm
```

Логи должны быть понятными и пригодными для диагностики.

---

# 14. State-файлы

Временные данные и служебная информация сохраняются в:

```
/var/lib/lsm
```

State-файлы используются для предотвращения повторной отправки уведомлений и хранения состояния модулей.

---

# 15. Runtime-файлы

Во время работы проекта используются:

```
/run/lsm
```

Здесь размещаются lock-файлы и временные данные.

---

# 16. Автоматизация

Для выполнения задач используются исключительно:

- systemd services;
- systemd timers.

Использование cron не планируется.

---

# 17. Установщик

Установщик должен:

- проверять совместимость системы;
- устанавливать зависимости;
- позволять выбрать необходимые модули;
- поддерживать готовые шаблоны конфигурации;
- поддерживать ручную настройку;
- создавать службы и таймеры;
- выполнять финальную проверку.

Повторный запуск установщика должен быть безопасным.

---

# 18. Обновление

Обновление должно сохранять:

- пользовательские настройки;
- секреты;
- историю;
- state-файлы.

---

# 19. Удаление

Удаление должно:

- удалить программу;
- удалить службы;
- удалить таймеры;
- удалить временные файлы.

Пользователь должен иметь возможность сохранить конфигурацию.

---

# 20. Требования к разработке

Все новые компоненты должны:

- быть модульными;
- иметь понятную структуру;
- использовать общую библиотеку проекта;
- поддерживать единый стиль логирования;
- использовать единый механизм уведомлений.
